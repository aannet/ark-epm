## 1. Objectif & Périmètre ⚠️

**Ce que cette feature fait :**

F-02 installe la couche i18n du frontend ARK. À l'issue de cette phase, toutes les strings visibles des composants F-01 et FS-01 sont externalisées dans un fichier `fr.json` unique, `react-i18next` est configuré et initialisé, et la convention `t('key')` est en place pour toutes les Feature-Specs suivantes. OpenCode utilisera ce fichier comme référence pour chaque nouveau composant généré.

Analogie : F-01 a posé les prises électriques. F-02 installe le tableau de distribution des labels — chaque pièce (feature) y branche ses libellés sans les câbler en dur dans les murs.

**Ce qu'elle ne fait pas (hors périmètre) :**

- Pas de sélecteur de langue dans l'UI — P2
- Pas de fichier `en.json` ni autre langue — P2
- Pas de lazy loading des traductions — inutile avec une langue unique
- Pas de traduction des messages d'erreur backend (renvoyés en anglais par NestJS, gérés côté frontend via des clés i18n mappées)
- Pas de RTL — hors périmètre MVP
- Pas de formatage de dates/nombres i18n (Intl natif suffisant en MVP)

## 3. Architecture i18n ⚠️

### 3.1 Structure des fichiers cible

```
frontend/src/
├── i18n/
│   ├── index.ts              ← Configuration i18next — importé en side-effect dans main.tsx
│   └── locales/
│       └── fr.json           ← Source de vérité de toutes les strings FR — voir §3.3
```

### 3.2 `i18n/index.ts`

```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import fr from './locales/fr.json';

i18n.use(initReactI18next).init({
  lng: 'fr',
  fallbackLng: 'fr',
  resources: {
    fr: { translation: fr },
  },
  interpolation: {
    escapeValue: false, // React gère déjà le XSS
  },
});

export default i18n;
```

## 4. Règles de Mise en Œuvre Critiques ⚠️

### RM-01 : Import side-effect dans `main.tsx` — avant tout

```typescript
// frontend/src/main.tsx
import '@/i18n';  // ← TOUJOURS en premier, avant les imports React/MUI
import React from 'react';
import ReactDOM from 'react-dom/client';
// ...
```

> L'import doit précéder tous les autres pour que `t()` soit disponible dès le premier rendu. Sans cela, les composants qui appellent `useTranslation()` au montage reçoivent une chaîne vide sur le premier cycle.

### RM-02 : Convention `t()` — jamais de string visible en dur dans les composants

```typescript
// ✅ Correct
import { useTranslation } from 'react-i18next';
const { t } = useTranslation();
<Button>{t('common.actions.save')}</Button>

// ❌ Interdit — refusé en code review
<Button>Enregistrer</Button>
```

> Exception acceptée : les strings purement techniques non visibles par l'utilisateur (attributs `aria-label` sur des icônes, noms de classes CSS, valeurs d'enum envoyées à l'API). En cas de doute, externaliser.

### RM-03 : Ajout de clés dans `fr.json` — en même temps que le composant

Chaque Feature-Spec suivante (FS-02 à FS-11) ajoute sa propre section dans `fr.json` lors de l'implémentation. Ne jamais créer un composant avec des strings et laisser l'ajout des clés pour plus tard.

Structure de section par feature :

```json
{
  "domains": {
    "list": { ... },
    "form": { ... }
  },
  "applications": {
    "list": { ... },
    "detail": { ... },
    "form": { ... }
  }
}
```

### RM-04 : Clés manquantes — comportement attendu

`i18next` en mode `fallbackLng: 'fr'` affiche la clé brute si la traduction est manquante (ex : `"domains.list.title"`). C'est le comportement souhaité en développement — visible immédiatement, sans crash.

> Ne pas utiliser `defaultValue` dans les appels `t()` — les clés manquantes doivent être détectées visuellement, pas masquées.

### RM-05 : Pas de `Trans` component en P1

Le composant `<Trans>` de react-i18next (pour les strings avec balises HTML imbriquées) est interdit en P1 — la complexité n'est pas justifiée. Si une string nécessite du markup riche, la décomposer en plusieurs appels `t()`.

## 5. Comportement Attendu par Cas d'Usage

**Nominal :**
- Tous les labels de la `Sidebar` sont issus de `t('nav.*')`
- `NotFoundPage` affiche `t('errors.notFound.title')` et `t('errors.notFound.description')`
- `UnauthorizedPage` affiche `t('errors.unauthorized.title')` etc.
- `ForbiddenPage` affiche `t('errors.forbidden.title')` etc.
- `ErrorBoundary` affiche `t('errors.unexpected.title')` + détail technique masqué hors dev
- `ConfirmDialog` utilise `t('common.confirmDialog.confirmLabel')` et `t('common.confirmDialog.cancelLabel')` par défaut, surchargeable via props
- `EmptyState` utilise `t('common.emptyState.title')` et `t('common.emptyState.description')` par défaut, surchargeable via props
- `LoginPage` affiche tous ses labels via `t('auth.login.*')`

**Clé manquante :**
- Si une clé est absente de `fr.json`, i18next affiche la clé brute — détectable visuellement immédiatement

## 8. Contraintes Techniques

- **`react-i18next` version :** `^14.x` (compatible React 18)
- **Pas de backend i18n :** Les messages d'erreur renvoyés par NestJS (ex : `"Domain not found"`) restent en anglais en base — la traduction se fait côté frontend en mappant le message ou le code HTTP vers une clé `t()`.
- **TypeScript strict :** Activer le typage des clés i18n est optionnel en P1 (nécessite `i18next-resources-to-backend` ou un plugin de génération de types). À évaluer en P2 si `fr.json` grossit significativement.
- **Alias `@/i18n` :** L'alias `@/` configuré en F-01 couvre déjà ce chemin — aucune modification de `vite.config.ts` nécessaire.
- **Aucun provider JSX supplémentaire :** `react-i18next` fonctionne via contexte global initialisé à l'import — pas de `<I18nextProvider>` à ajouter dans `main.tsx`.
